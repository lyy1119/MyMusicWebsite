
### 一、 项目需求详细定义

#### 1.1 用户与角色权限系统

项目主要面向个人及小规模群体，采用严格的准入机制，不开放公共注册，但保留注册功能的开发拓展，各个用户拥有唯一id可作为昵称和登录方式。

* **管理者 (Admin)**：
  * 管理用户账号（创建、删除）。
  * 管理全站上传内容，出于内容安全考虑，可修改上传音乐的备注或直接删除内容。
  * 设定“使用者”角色的上传限额（设定为 `-1` 代表无限制）。
  * 配置全站默认的自定义 UI 样式（用于同好会等特定圈子展示）。
  * 项目初始化部署时，系统自动随机生成默认 Admin 的 ID 和密码。

* **使用者 (User)**：
  * 通过邮箱号或者id登录，可修改id。
  * 可以手动上传音视频文件。
  * 可以创建及管理个人歌单，接受他人分享。
  * 支持自定义个人的 UI 样式。

* **听众 (Audience)**：
  * 通过邮箱号或者id登录，不可修改id。
  * 仅能以歌单形式听取他人分享的内容，不支持上传和独立建单。
  * 支持自定义个人的 UI 样式。

#### 1.2 认证与安全生命周期

系统实施以邮箱为核心的最高安全级别鉴权流。

* **账户创建与引导**：由管理员手动创建账号后，系统自动通过邮件发送包含系统引导提示的通知至用户邮箱，此时账号处于无密码、无 TOTP 的初始状态。
* **首次登录与初始化**：用户初次访问必须且仅能通过“邮箱验证码”进行登录。验证成功后，用户可以在设置页自行设定常规登录密码或绑定 TOTP（时间同步动态密码）。
* **日常登录支持**：支持密码登录、TOTP 登录，以及无密码登录（无密码登录仅依赖邮箱验证码或 TOTP）。
* **敏感操作二次校验**：邮箱验证码具有最高安全级别，任何修改密码、删除密码、修改或解绑 TOTP 的操作，均强制要求通过邮箱验证码进行二次验证。

#### 1.3 内容上传与媒体管理

* **音视频分离**：上传视频时，系统提供并默认开启自动分离音频的选项（方便同时收录 MV 与歌曲）。
* **自动向下转码**：上传高音质音频或从视频提取的音频时，后台自动尝试转码并保存为适合网页播放的低音质版本。音质阶梯规定如下：
  * 流畅：96 kbps - 128 kbps AAC 
  * 高品质：192 kbps - 320 kbps AAC 
  * 无损：600 kbps - 1000 kbps FLAC (16-bit / 44.1kHz) 
  * HI-RES：2000 kbps - 4000 kbps 及以上 FLAC (24-bit / 48kHz - 192kHz) 
* **智能解析与封面处理**：自动尝试解析音频的封面与 meta 信息（支持手动替换选项）。封面图片自动处理并保存为 512x512（大图展示）和 64x64（小图展示）两种尺寸的文件。
* **文件关联系统**：视频需与分离出的各音质音频保持关联，音频需与歌词文件（支持手动上传关联）及封面图片保持关联。
* **去重与垃圾回收 (GC)**：
  * 基于文件哈希值（如 SHA-256）计算，若重复则实现秒传，直接复用底层现有内容以节约空间。
  * 底层物理文件维护 `Ref_Count`（引用计数）字段，当上层逻辑实体（歌曲/视频）被删除或替换导致引用计数降为 0 时，系统自动清理真实物理文件。

#### 1.4 分享交互机制

* **分享类型**：提供“分享码定向分享”与“全局共享”两种模式。全局共享内容对所有已登录用户可见。
* **游客限制**：暂不对非注册用户（未登录访客）提供分享查看功能，但系统底层保留后续拓展该能力的接口。

#### 1.5 前端界面与 UI 引擎

* **核心页面**：主页（唯一允许任何人公开访问的欢迎页面）、登录页、管理员管理页、使用者上传页、音乐/MV 选择库、音乐/MV 播放页（包含歌词展示）。
* **PWA 与轻量化**：采用 PWA（Progressive Web Apps）技术支持用户直接“安装”为桌面/移动端应用。页面 DOM 与 CSS 追求极简，最小化 JS 依赖，以保障低性能设备的流畅加载与兼容。
* **受限的自定义主题系统**：
  * 内置基础的“白色”与“黑色”两套主题样式。
  * 提供主题自定义能力，允许用户调整背景、颜色、元素布局与外观。
* **安全隔离**：只允许上传 CSS 文件和补充性质的 JSON 配置文件，从根本上禁止上传 JS 脚本以杜绝 XSS 风险。
* **生效范围**：登录页等公共可访问网页保持全局隔离，不受任何自定义样式的覆盖和干扰。

#### 1.6 部署架构与底层设计

* **数据库设计原则（实体与介质分离）**：
  * 采用分离式设计：将业务逻辑实体（`Songs` 歌曲元数据表）与底层物理介质（`Files` 物理文件表）拆分。
  * 通过 `Media_Relations`（媒体关联表）将多个不同音质的音频文件、MV 视频文件、歌词等关联至同一个逻辑歌曲 ID 下。
* **部署模式 (现场编译部署)**：采用 Docker-First 理念，默认进行现场编译（Build from source），不提供打包好的臃肿镜像。
* **后端**：Go 语言单文件可执行程序，通过 Alpine 基础镜像部署。镜像构建阶段通过 `apk add ffmpeg` 安装音视频处理底层依赖。
* **前端**：React 编译为静态 HTML/JS/CSS 文件，通过 Nginx 镜像部署托管。
* **路由代理**：默认前后端同服务器部署，Nginx 处理前端路由并通过统一前缀（如 `/api`）反向代理至后端，同时保留指定后端地址以实现完全分离部署的选项。
* **环境参数注入 (`Environment`)**：所有核心配置（如 SMTP 邮件服务器参数、自定义音频物理存储路径等）均要求能通过 Docker 的环境变量直接指定。

### 二、 涉及的技术栈总览

根据上述需求与架构设计，项目的技术栈清单如下：

#### 2.1 前端技术栈 (Frontend)

* **核心框架**：React.js 
* **应用特性**：PWA (Progressive Web App) + Service Worker 
* **样式与状态**：纯原生 CSS 变量注入（用于主题引擎）、受控的 JSON 配置解析、轻量级状态管理（如 Zustand 或 Context API） 
* **媒体播放内核**：HTML5 原生 `<audio>` / `<video>` 结合 Range Requests 缓冲控制 
* **部署托管**：Nginx (Alpine) 处理静态资源与反向代理 

#### 2.2 后端技术栈 (Backend)

* **核心语言**：Go (Golang) - 负责高并发 API 响应、数据库读写、文件切片流传输 
* **数据库**：SQLite (默认嵌入式使用，通过 ORM 或接口层保留向 MySQL/PostgreSQL 迁移的拓展性) 
* **媒体处理引擎**：FFmpeg (通过 `os/exec` 在 Go 后端异步调度调用) 
* **认证机制**：基于 JWT/Session 的会话管理、TOTP 算法库 (Time-based One-Time Password)、SMTP 邮件发送服务 

#### 2.3 运维与基础设施 (DevOps & Infra)

采用容器化部署，但是也提供可执行程序直接部署的可能。

* **容器化**：Docker & Docker Compose 
* **基础镜像**：Alpine Linux (用于运行 Go 可执行文件并安装 FFmpeg 依赖) 
* **存储机制**：Docker Volume 宿主机目录挂载 (物理分离媒体文件与数据库 `.db` 文件)
